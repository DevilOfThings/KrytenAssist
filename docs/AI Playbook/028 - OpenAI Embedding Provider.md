# Prompt 028 – OpenAI Embedding Provider

## Goal

Introduce an OpenAI-backed embedding provider for the Avalonia desktop client while preserving the existing offline embedding architecture.

Prompt 026 introduced the `IEmbeddingService` abstraction and a deterministic offline implementation.
Prompt 027 used that abstraction to add hybrid keyword and semantic search.

This prompt adds a real embedding provider behind the same abstraction so the semantic search pipeline can become more meaningful. Because the existing search runs as the user types, this prompt must also add the minimum request-control safeguards needed to prevent an OpenAI request for every keystroke and every candidate prompt.

This is a provider prompt only.

No UI redesign, chat interface, memory system, vector database, embedding persistence, background indexing, or prompt store schema changes should be added in this prompt.

---

## Why This Prompt Exists

The Avalonia client now has a complete offline search pipeline:

- Keyword search identifies candidate prompts.
- Embeddings are generated through `IEmbeddingService`.
- Cosine similarity ranks matching prompts.
- `FilteredPromptCards` presents the ranked results.

Currently, embeddings are generated by `DeterministicEmbeddingService`. This is useful for offline development and architectural testing, but it does not provide meaningful semantic understanding.

This prompt introduces an OpenAI-backed implementation while keeping the deterministic implementation available as a fallback and development option.

The important architectural goal is that `MainWindowViewModel` should not know or care which embedding provider is being used.

---

## Scope

Implement only OpenAI embedding provider support.

## Architecture

The implementation should preserve the existing abstraction boundary:

- `IEmbeddingService` remains the public embedding abstraction.
- `DeterministicEmbeddingService` remains available for offline development.
- `OpenAIEmbeddingService` implements `IEmbeddingService`.
- Provider selection is handled through configuration and Dependency Injection.
- The hybrid search architecture from Prompt 027 remains unchanged.
- The ViewModel should not reference OpenAI-specific types.
- Live search should debounce requests and cancel obsolete searches.
- Prompt embeddings should be cached in memory for the lifetime of the ViewModel and invalidated when prompt data changes.
- Runtime OpenAI failures should fall back to deterministic embeddings while keeping the application fully usable.
- Embedding provider status should be exposed through an abstraction rather than OpenAI-specific types.

## Design Principles

- Preserve offline-first development support.
- Keep OpenAI details behind `IEmbeddingService`.
- Use configuration rather than hard-coded provider selection.
- Avoid issuing remote embedding requests on every keystroke.
- Cancel obsolete searches so stale results cannot overwrite newer results.
- Cache prompt embeddings in memory to avoid repeatedly embedding unchanged prompt templates.
- Distinguish startup configuration failures from runtime provider failures.
- Fail fast for invalid startup configuration.
- Degrade gracefully for runtime provider failures.
- Do not store API keys in source control.
- Do not change the prompt JSON schema.
- Do not change the semantic search pipeline.
- Keep the implementation replaceable so future local or alternative providers can be added.

### In Scope

- Add an embedding options model.
- Add configuration for embedding provider selection.
- Add configuration for the OpenAI embedding model.
- Add an OpenAI embedding service implementing `IEmbeddingService`.
- Register the correct embedding provider through Dependency Injection.
- Add a short debounce before remote semantic ranking begins.
- Cancel an in-progress semantic search when the search text changes again.
- Cache generated prompt embeddings in memory and clear the cache after prompt data changes.
- Add a resilient embedding service that wraps the configured provider.
- Fall back to deterministic embeddings when OpenAI requests fail at runtime.
- Expose a non-fatal status message indicating when fallback mode is active.
- Preserve `DeterministicEmbeddingService` as the default or fallback provider.

- Confirm the existing semantic search pipeline still builds and runs.

## Implementation Notes

- Add an options class such as `EmbeddingOptions`.
- Include a provider setting such as `Deterministic` or `OpenAI`.
- Include an OpenAI embedding model setting.
- Prefer a small default model such as `text-embedding-3-small` unless a different model is deliberately chosen.
- Read the OpenAI API key from the `OPENAI_API_KEY` environment variable.
- Store the API key only as an operating system environment variable.
- Do not store the API key in appsettings.json or commit it to source control.
- If OpenAI is selected but the API key is missing, fail fast during application startup with a clear configuration error.
- Do not hard-code secrets in source files or place them in `appsettings.json`.
- Keep `Deterministic` as the default provider.
- Keep `DeterministicEmbeddingService` registered when the selected provider is deterministic.
- If `OpenAI` is explicitly selected but its API key is missing, fail with a clear configuration error rather than silently changing provider.
- The OpenAI provider should return the embedding values as an `EmbeddingVector`.
- Add a short search debounce, approximately 300–500 milliseconds, before remote embedding requests begin.
- Use cancellation so a superseded search cannot update `FilteredPromptCards` after a newer search.
- Cache prompt embeddings by prompt identity and searchable content for the lifetime of the ViewModel.
- Clear or refresh cached prompt embeddings whenever prompts are loaded or saved.

## Suggested Configuration Shape

A simple options class is sufficient:

```csharp
public sealed class EmbeddingOptions
{
    public string Provider { get; init; } = "Deterministic";
    public string Model { get; init; } = "text-embedding-3-small";
}
```

A future prompt can expand this if additional providers are added.

---

## Configuration

Example `appsettings.json`:

```json
{
  "Embedding": {
    "Provider": "Deterministic",
    "Model": "text-embedding-3-small"
  }
}
```

The default provider should remain `Deterministic` so the application can be developed and tested entirely offline.

To enable OpenAI embeddings, change the provider to `OpenAI` and define the `OPENAI_API_KEY` environment variable before starting the application.

The API key must never be committed to source control or stored in `appsettings.json`.

Example (macOS):

```bash
export OPENAI_API_KEY="sk-proj-..."
```

Verify the variable is available:

```bash
echo $OPENAI_API_KEY
```

If `OpenAI` is selected but `OPENAI_API_KEY` is not defined, the application should fail during startup with a clear configuration error rather than silently falling back to the deterministic provider.

## Lessons Learnt

- Validate configuration during application startup rather than waiting for the first embedding request.
- Distinguish startup configuration failures from runtime provider failures.
- Debouncing, cancellation and caching are essential when introducing remote AI services into an interactive desktop application.
- Preserve offline development by keeping deterministic implementations available.
- Runtime provider failures should degrade gracefully rather than silently breaking application behaviour.
- Provider resilience belongs behind the `IEmbeddingService` abstraction rather than inside the ViewModel.
- User-visible status messages make AI provider failures understandable without exposing implementation details.
