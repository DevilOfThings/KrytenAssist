

# Prompt 026 – Avalonia Embedding Service

## Goal

Introduce the first embedding service foundation for the Avalonia desktop client.

This prompt adds a local embedding abstraction that can convert prompt template text into numeric vectors. These vectors will later support semantic search, similarity ranking, prompt recommendations, and AI-assisted organisation.

This is a foundation prompt only.

No semantic search UI, OpenAI integration, vector database, embedding persistence, prompt ranking, background indexing, or cloud synchronisation should be added in this prompt.

---

## Why This Prompt Exists

Prompt 025 introduced offline keyword search using a filtered collection:

- `PromptCards` remains the source of truth.
- `FilteredPromptCards` is the UI projection.

That pattern gives the desktop client a safe place to add richer search behaviour later.

Before semantic search can be introduced, the application needs a small, testable embedding service that can produce vector representations of text without changing the UI or persistence model.

This prompt introduces that capability in isolation.

The aim is to prepare the architecture for semantic search while preserving the existing offline search experience unchanged.

---

## Scope

Implement only the embedding service foundation.

## Architecture

The implementation should continue following the existing Avalonia offline-first architecture:

- The View owns layout and data binding.
- The ViewModel owns UI state and user-facing behaviour.
- The JSON prompt store remains the prompt persistence mechanism.
- Embeddings are generated by a service abstraction.
- Dependency Injection resolves the embedding service.
- No UI behaviour should depend directly on a concrete embedding implementation.

The embedding service should be designed so that a future prompt can replace or supplement the local implementation with an OpenAI-backed implementation.

## Design Principles

The implementation should favour safe architecture over visible features.

- Add the abstraction before adding semantic search.
- Keep the first implementation deterministic and offline.
- Avoid introducing external API dependencies.
- Avoid storing embeddings until there is a clear indexing strategy.
- Keep generated vectors simple and predictable for testing.
- Preserve all current prompt editor, category and search behaviour.

### In Scope

- Create an embedding result model.
- Create an `IEmbeddingService` abstraction.
- Add a local deterministic embedding implementation.
- Generate embeddings from plain text input.
- Register the embedding service with Dependency Injection.
- Add basic verification that embeddings can be generated.
- Keep the implementation fully offline.

### Out of Scope

- OpenAI API integration.
- Semantic search.
- Replacing the existing keyword search.
- Ranking prompt cards by vector similarity.
- Cosine similarity calculations unless needed for a minimal internal test.
- Embedding persistence.
- JSON schema changes.
- SQLite.
- Vector databases.
- Background indexing.
- UI changes.
- API synchronisation.
- Cloud storage.
- AI chat integration.

## Implementation Notes

- Add a small model to represent an embedding vector.
- Prefer a read-only collection or array of numeric values for the vector values.
- Keep the embedding model independent from `PromptCardModel` for now.
- Create an `IEmbeddingService` interface with an asynchronous method such as `GenerateEmbeddingAsync`.
- The interface should accept plain text and return an embedding model.
- The first implementation should be deterministic so that the same input always produces the same vector.
- The first implementation does not need to be intelligent; it only needs to prove the shape of the architecture.
- A simple hashing, character bucket, token bucket, or fixed-size vector approach is acceptable.
- Normalise null, empty or whitespace input safely.
- Do not introduce OpenAI packages or API keys in this prompt.
- Do not modify the JSON prompt store schema.
- Do not change `FilteredPromptCards` behaviour.
- Keep current keyword search as the only visible search behaviour.

## Suggested Embedding Shape

A simple deterministic implementation is sufficient for this prompt.

For example:

- Use a fixed vector size such as 32 or 64.
- Split the input into lowercase tokens.
- Hash each token into a bucket.
- Increment the corresponding vector value.
- Optionally normalise the vector length.

This is not intended to produce high-quality semantic meaning.

It exists only to make later semantic search work easier to implement and test.

## Lessons Learnt

- Introduce architectural abstractions before implementing user-facing features.
- A deterministic local embedding implementation provides a safe foundation for future semantic search.
- `string.GetHashCode()` should not be used when deterministic behaviour is required across application runs because its output is intentionally randomised by .NET.
- A stable hashing algorithm such as FNV-1a provides predictable, repeatable bucket allocation suitable for offline deterministic embeddings.
- Registering services with Dependency Injection before first use keeps the composition root clean and simplifies future feature development.
- Small, independently verifiable prompts continue to produce maintainable architecture while reducing implementation risk.
