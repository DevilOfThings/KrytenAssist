### Result

Successfully implemented the first provider-independent tools framework for the Avalonia desktop client.

The AI conversation service can now:

- discover registered application tools
- expose tool definitions to OpenAI
- receive provider tool-call requests
- translate provider calls into `ToolInvocation`
- execute tools through `IToolRegistry`
- return `ToolResult` values to the provider
- continue the conversation until a final natural-language response is produced

The ViewModel remains unaware of individual tools and provider-specific tool-call types.

Implemented three deterministic built-in tools:

- `get_current_date_time`
- `calculate`
- `get_application_info`

Added a central `ToolRegistry` responsible for:

- discovering registered `ITool` implementations
- exposing provider-independent definitions
- resolving tools by stable case-sensitive name
- rejecting duplicate registrations
- executing known tools
- returning controlled failures for unknown tools
- propagating cancellation tokens

Added a bounded tool-call loop using configurable `MaxToolIterations`.

Prompt 030 memory behaviour remains intact:

- only the user message and final assistant response are committed
- intermediate tool-call messages are not stored
- raw tool arguments and results are not stored
- failed and cancelled conversations are not committed

Introduced `IClock` and `SystemClock` so time-dependent behaviour can be tested deterministically.

Added a dedicated Avalonia test project and focused tests for the registry and all built-in tools.

## Lessons Learnt

- Provider-independent tool abstractions (`ITool`, `ToolInvocation`, `ToolDefinition` and `ToolResult`) keep the application architecture independent of any specific AI provider.
- The conversation service should be responsible for translating between provider-specific SDK models and the application's provider-independent tool models.
- Registering tools through dependency injection and discovering them via `IEnumerable<ITool>` allows new capabilities to be added without modifying the conversation service.
- A central `ToolRegistry` provides a single, testable location for tool discovery, validation and execution while preventing duplicate tool registrations.
- Limiting tool execution with a configurable `MaxToolIterations` prevents accidental infinite tool-call loops.
- Conversation memory should only persist successful user messages and final assistant responses. Intermediate tool-call messages and tool results should remain transient.
- Introducing the `IClock` abstraction with a `SystemClock` implementation made time-dependent behaviour deterministic and straightforward to unit test.
- Structured tool results should be validated by parsing their JSON content rather than comparing raw serialized strings, making tests more resilient to formatting changes.
- Dedicated unit tests for every built-in tool significantly increased confidence in the provider-independent tool framework and made future refactoring safer.
- Keeping dependency injection registrations inside extension methods keeps `Program.cs` concise and makes the application startup easier to maintain.
- Implementing features in small, buildable increments with frequent `dotnet build` and `dotnet test` verification reduced debugging time and simplified issue isolation.
- Referring to the target file together with approximate line numbers and nearby code anchors made manual code updates much more reliable during pair-programming sessions.
