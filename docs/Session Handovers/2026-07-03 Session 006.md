# Database Changes

### Added

- EF Core Design package
- Design-time `DbContext` factory
- Initial migration
- Model snapshot
- Automatic migration support

### Updated

`DatabaseInitializer`

Changed from:

```csharp
context.Database.EnsureCreated();
```

to:

```csharp
context.Database.Migrate();
```

The application now creates the database (if required) and automatically applies pending migrations during startup.

---

# Migration Workflow

The project now supports the standard EF Core workflow:

```bash
dotnet ef migrations add <MigrationName>
dotnet ef database update
```

Future schema changes can now be tracked safely through source control.

---

## Existing Database

When switching from `EnsureCreated()` to `Migrate()`, integration tests initially failed with:

```text
SQLite Error 1:
table "PromptCards" already exists
```

Cause:

Existing development databases created using `EnsureCreated()` do not contain the `__EFMigrationsHistory` table.

Resolution:

Deleted the local SQLite database and allowed EF Core to recreate it using the generated migration.

# Lessons Learnt

- `EnsureCreated()` should not be used once migrations are introduced.
- Existing databases created with `EnsureCreated()` must be recreated before switching to migrations.
- A design-time `DbContext` factory keeps EF tooling concerns isolated within Infrastructure.
- Database schema evolution is now version controlled.
- Integration tests provided confidence that introducing migrations did not change the API contract.
