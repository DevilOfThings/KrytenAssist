# Completed Since Previous Session

## Prompt 011 – Validation ✅

### Completed

Introduced FluentValidation for request models.

### Added

- CreatePromptCardRequestValidator
- UpdatePromptCardRequestValidator

### Key Decisions

- Validation belongs in the Application layer.
- FluentValidation selected as the validation framework.
- Validation occurs before application use cases execute.
- Invalid requests return HTTP 400 using `Results.ValidationProblem(...)`.
- Business logic remains inside the use cases.
- Manual validation removed from CreatePromptCard.

---

## Prompt 012 – Application Dependency Injection Cleanup ✅

### Completed

Moved dependency injection registration to the layer that owns each service.

### Created

```text
KrytenAssist.Application/DependencyInjection.cs
```

### Updated

- Program.cs
- Infrastructure/DependencyInjection.cs

### Key Decisions

- Application owns registration of Application services.
- Infrastructure owns registration of Infrastructure implementations.
- Validators are registered using assembly scanning.
- Program.cs remains the composition root.
- No functional behaviour changed.

# Development Workflow

For every prompt:

1. Discuss the design.
2. Agree the architecture.
3. Create the AI Playbook prompt.
4. Implement one file at a time.
5. Build after each significant step.
6. Test functionality.
7. Update AI Playbook.
8. Update Roadmap.
9. Commit to Git.
10. Create a Session Handover when ending a session.

---

# Lessons Learnt

- Small architectural prompts significantly improve maintainability.
- Build after every meaningful change to minimise debugging.
- Dependency registration should be owned by the layer that owns the services.
- FluentValidation assembly scanning avoids manual DI updates as validators grow.
- Session Handovers should only be created at the end of a session—not after every prompt.
- Displaying progress as **Step X of Y** makes long implementation prompts much easier to follow.
